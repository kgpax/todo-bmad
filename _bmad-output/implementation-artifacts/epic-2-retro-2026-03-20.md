## Project Lead reflections (Kevin)

**What went well**

- The team was **better at splitting new requirements into separate stories** instead of absorbing scope silently.

**What went less well**

- **Process discipline slipped sometimes** — key process steps were skipped on occasion.
- **Branch-before-implementation** was missed for **two stories** (work proceeded before the feature branch existed).
- **Backend production path** — `npm run build` then `npm run start` for the backend **should have been verified from the first Epic 1 story**; it was **not checked early**, and issues surfaced later. Root `start` and production Lighthouse work landed in Story 2.6, which corrected the gap for the full stack but does not change the lesson: **compiled backend startup should be a first-class gate for foundation work.**

## Action items

### Process

1. **Story kickoff checklist — branch first**  
   Before the first implementation commit on a story: **feature branch exists** and matches the story key.  
   **Owner:** Bob (Scrum Master) with Amelia (Developer Agent)  
   **Success criteria:** No story is marked ready-for-review or done unless the branch was created before implementation; **two misses in Epic 2** are treated as the baseline to improve from, not the norm.

2. **Non-negotiable process steps on story closeout**  
   Align with `project-context.md` / Definition of Done: explicitly confirm **lint, build, unit tests (with coverage gate), E2E, Lighthouse** where applicable, plus **branch discipline** (item 1).  
   **Owner:** Bob (Scrum Master), Dana (QA Engineer)  
   **Success criteria:** Story completion notes explicitly state each gate or document a deliberate exception with owner approval.

### Technical / verification

3. **Backend production smoke: `build` + `start`**  
   For any work that touches the backend (and as ongoing hygiene): after `npm run build` in `packages/backend`, **`npm run start`** (or documented equivalent) must succeed against compiled output. Consider documenting this in README / `project-context.md` as a **foundation verification** step, not only a Story 2.6 concern.  
   **Owner:** Charlie (Senior Dev) with Winston (Architect)  
   **Success criteria:** Documented steps and a repeatable check; failures block merge or are flagged before story completion.

4. **Brownfield note for future scaffolding epics**  
   When standing up a repo similar to Epic 1, **first stories** validate: `packages/backend` build + `node` on `dist`, `packages/frontend` build + `next start`, and **root** `build` + `start` once the root script exists.  
   **Owner:** Winston (Architect)  
   **Success criteria:** Architecture or project-context captures this so it is not rediscovered only in a late epic.
